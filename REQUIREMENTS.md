# 上光廠內部網站系統需求規格 v1（討論稿）

> 目的：先確認需求，不急著開發。此文件只定義畫面與流程，後續再進入資料庫與程式實作。

## 1. 整體流程

1. 使用者打開網站。
2. 顯示公司首頁（顯示公司名稱、帳號、密碼欄位）。
3. 登入後進入「導航頁」。
4. 導航頁有三個功能框：
   - 工單作業系統
   - 客戶資料系統
   - 財經資料系統
5. 點擊不同框進入對應子系統。
6. 只有「財經資料系統」需要再次輸入密碼（次級密碼保護）。

## 2. 權限與登入

- 目前先不做複雜角色權限。
- 基本登入：首頁帳號密碼。
- 僅財經資料系統需要額外密碼驗證。
- 工單與客戶資料系統登入後可直接進入。

## 3. 導航頁（首頁後的主畫面）

- 以三個明顯框塊（卡片）作為入口。
- 每個框塊可點擊。
- 框塊是「導航概念」，不直接塞全部內容。

### 3.1 三個框塊

1. 工單作業系統
2. 客戶資料系統
3. 財經資料系統（點擊時跳出需輸入密碼）

## 4. 工單作業系統需求

## 4.1 新增工單欄位

- 工單編號：保留空格手動輸入（不自動編號）
- 日期
- 上游客戶（下拉，必填）
- 下游客戶（下拉，必填）
- 張數
- 尺寸：長 × 寬
- 尺寸單位：可選（例如 mm / cm / inch）
- 上光種類：
  - 下拉預設選項：A光、B光
  - 可自由輸入新種類並加入選單
- 總價（先只放總價欄位，不做單價/自動計算）

> 客戶名稱不做獨立欄位，工單中的客戶資訊以「上游客戶 + 下游客戶」組成。

## 4.2 尺寸與單位問題（你的困擾對應方案）

為了處理有時單位不同，新增工單時採「拆欄位」：

- 長（數字）
- 寬（數字）
- 單位（下拉：mm/cm/inch，可再擴充）

好處：

- 不同單位可共存，不會混亂。
- 之後如果要自動換算也做得到。
- 查詢與報表統計更容易。

## 4.3 工單修改紀錄（重點）

你要求：工人可以改價格，但要可追蹤誰、哪台電腦、什麼時間修改。

因此需有「修改稽核紀錄」：

- 工單 ID
- 修改欄位（例如總價）
- 修改前值 / 修改後值
- 修改時間
- 操作者帳號
- 電腦識別資訊（至少 IP + 裝置資訊）

## 5. 客戶資料系統需求

- 維護客戶主檔。
- 客戶需可標記為：
  - 上游
  - 下游
  - 兩者皆可
- 工單新增時的「上游/下游下拉選單」來源即為此處客戶資料。

## 6. 財經資料系統需求（先期）

- 入口需二次密碼。
- 細節功能先不展開，待下一版訪談定義。

## 7. 第一版（MVP）建議範圍

優先順序：

1. 登入頁 + 導航頁（三框）
2. 工單新增/列表/編輯
3. 客戶主檔（支援上下游下拉來源）
4. 價格修改稽核紀錄
5. 財經入口的二次密碼

## 8. 下一步待你確認的項目

1. 公司名稱顯示文字（登入頁標題）。
2. 財經系統二次密碼規則（固定密碼 or 每人不同）。
3. 工單狀態是否要加（待處理、進行中、完成）。
4. 客戶資料是否要聯絡人/電話/地址。
5. 稽核紀錄是否要可匯出 Excel。

## 9. 建議再補強的規格（避免後續重工）

以下是目前最值得先補進 v1 的內容：

1. 工單狀態欄位（建議固定值）
   - 待處理 / 製作中 / 已完成 / 已取消
   - 讓現場可以快速知道卡在哪個環節。

2. 價格修改需填原因（建議）
   - 除了記錄誰、何時、哪台電腦，建議再加「修改原因」。
   - 例如：客戶追加、尺寸修正、重工折讓。

3. 下拉選單怎麼管理（白話版）
   - 例如上光種類原本有「A光」，之後不用了：
     - 不要刪掉 A光。
     - 只把 A光 設成「停用」。
   - 這樣舊工單裡原本選 A光 的資料還會完整，不會壞掉。
   - 上游/下游客戶下拉也用同一個做法：不用的客戶改停用，不直接刪除。

4. 財經系統二次密碼流程細節
   - 保留二次密碼即可，不做錯誤次數鎖定。
   - 仍建議把錯誤嘗試留稽核紀錄（誰、時間、IP）。

5. 舊系統資料搬遷與備份
   - 第一版就要定義「每日自動備份」。
   - 提供 Excel 匯出，避免再次被單一系統綁死。

6. 稽核查詢頁
   - 能依日期、工單號、操作者查詢價格變更紀錄。
   - 這會直接影響你日常追查效率。
